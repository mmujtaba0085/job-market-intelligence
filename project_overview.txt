1. WHAT THIS PROJECT IS
================================================================================

Job Market Intelligence is a self-hosted Python application that:

  - Collects public job postings from multiple free/paid APIs (no scraping).
  - Normalizes, deduplicates, and stores all postings in a local SQLite database.
  - Extracts skill signals from job descriptions (regex + taxonomy, no ML model
    required).
  - Computes weekly analytics: top skills, fastest-growing skills, emerging
    signals, co-occurrence pairs, remote trends, company breakdowns.
  - Generates weekly Substack-ready Markdown + HTML reports.
  - Exports curated job rows to Google Sheets (one spreadsheet per country).
  - Hosts a local BI dashboard (Flask) with filtering, charts, and data exports.
  - Tracks how many times each job posting is clicked via a Google Apps Script
    web app and records those clicks back to a central Tracker spreadsheet.
  - Uses AI (Groq / xAI Grok) for two tasks:
      a) Classifying staged jobs as "IT" or "Non-IT" before Sheets upload.
      b) AI-enhanced quality review of job records (title/location/remote fixes).

The project is designed to run locally on Windows (with a Python venv) and uses
only an SQLite file as its primary database — no Postgres, no Redis, no server
infrastructure is required to run the core pipeline.

5. DATA PIPELINE (END-TO-END FLOW)
================================================================================

STEP 1 — COLLECTION (src/collectors/)
  Each enabled collector runs collect(market) → list[JobRaw].
  JobRaw contains: source_id, url, fetched_at, raw_json or parsed_fields.

STEP 2 — NORMALIZATION (src/normalizer.py)
  normalize(job_raw) → JobNormalized or None (if critical fields missing).
  Computes four hashes, infers remote_type, parses dates, cleans strings.
  Calls title_normalizer.normalize_title() for canonical title.

STEP 3 — DEDUPLICATION + STORAGE (src/deduplicator.py + storage/db.py)
  Layer 1: url_hash match → update last_seen_at, skip insert (url_dup).
  Layer 2: canonical_hash match with new location → add location (location_added).
  Layer 2: canonical_hash match, same location → skip (canonical_dup).
  Otherwise: INSERT new job row (inserted).

STEP 4 — SKILL EXTRACTION (src/skill_extractor.py)
  For each newly inserted job, scan description_text against all terms in
  SKILL_TAXONOMY + SKILL_SYNONYMS (config/taxonomy.py).
  Pre-compiled regex patterns, word-boundary aware, longest-first matching.
  Each detected skill → SkillSignal → inserted into skills table.

STEP 5 — ANALYTICS (src/analytics/)
  compute_weekly_metrics()     — skill frequency + WoW growth + mover score
  assign_flags()               — emerging / declining flags
  compute_co_occurrence()      — skill pairs that appear together
  compute_temporal_trends()    — multi-week trends per skill
  compute_title_stats()        — job title distribution
  compute_category_stats()     — skill category breakdown
  compute_coverage_stats()     — remote%, sources breakdown, geo distribution

STEP 6 — REPORTS (src/reports/)
  generate_markdown_report()   — Jinja2 → outputs/{market}/{week}/report.md
  convert_to_html()            — report.md → report.html (Substack-paste-ready)
  export_top_skills CSV        — outputs/{market}/{week}/top_skills.csv
  export_growth_skills CSV     — + many other CSV exports
  generate_all_charts()        — matplotlib PNGs (remote split, skill trends)
  run.finish()                 — outputs/{market}/{week}/run_summary.json

STEP 7 — GOOGLE SHEETS UPLOAD (src/reports/google_sheets_export.py)
  Reads sheets_staging rows (status = 'staged' or 'pending').
  Groups by country → writes to each country's Google Spreadsheet.
  Auto-creates tabs per normalized_title if they don't exist.
  Each job row gets a tracking URL: Apps Script endpoint + encoded params.
  After upload, calls export_directory() to sync Tracker spreadsheet.

STEP 8 — AI STAGING CLASSIFICATION (src/ai/grok_staging.py)
  Reads pending rows from sheets_staging.
  Sends chunks of 25 rows to Groq API (llama-3.3-70b).
  Classifies each row as tab_bucket: "IT" or "Non-IT".
  Falls back to IT_KEYWORDS local matching if AI is unavailable.
  Multiple API keys rotated with per-key rate-limit tracking.

PIPELINES are coordinated by src/orchestrator.py:
  --mode weekly       = Steps 1-6 + optional Sheets push
  --mode ingest-only  = Steps 1-4 only
  --mode report-only  = Steps 5-6 only
  --mode crawl        = Continuous Findwork full-catalogue crawl
  --backfill          = Steps 5-6 for a date range from existing DB data

  description_hash TEXT
  title           TEXT
  company         TEXT
  country         TEXT
  location        TEXT
  remote_type     TEXT ("remote" | "hybrid" | "on-site" | "unknown")
  posted_date     TEXT (ISO date, nullable)
  salary_min      REAL (nullable)
  salary_max      REAL (nullable)
  currency        TEXT (nullable)
  raw_description TEXT
  normalized_title TEXT (added via migration)
  all_locations   TEXT (JSON array, for multi-location jobs)
  first_seen_at   TEXT (ISO datetime)
  last_seen_at    TEXT (ISO datetime)
  ingested_at     TEXT (ISO datetime)

skills
  id              INTEGER PK
  job_id          INTEGER FK → jobs
  market_id       TEXT
  raw_detected_skill TEXT
  normalized_skill   TEXT
  category           TEXT
  confidence_score   REAL (nullable)
  method             TEXT (default "regex_taxonomy")

weekly_metrics
  id              INTEGER PK
  market_id       TEXT
  week_start_date TEXT (ISO date, Monday)
  week_number     INTEGER
  skill_name      TEXT
  category        TEXT
  frequency       INTEGER
  growth_percentage REAL
  absolute_delta  INTEGER (this_week - prior_week count)
  mover_score     REAL (delta × log1p(frequency))
  emerging_flag   INTEGER (0/1)
  declining_flag  INTEGER (0/1)
  UNIQUE(market_id, week_start_date, skill_name)

sheets_staging
  id              INTEGER PK
  job_id          INTEGER FK → jobs
  override_title, override_normalized_title, ... (editable overrides)
  assigned_tab    TEXT (normalized_title / IT bucket)
  assigned_sheet  TEXT (country name)

  status          TEXT ("pending" | "staged" | "uploaded" | "approved" | "excluded")
  exclude_from_upload INTEGER (0/1)

  UNIQUE(job_id, assigned_sheet, assigned_tab)

sheets_targets
  id              INTEGER PK
  name            TEXT (e.g. "Canada Jobs", "UK Jobs")
  private_spreadsheet_id   TEXT (Google Sheets edit ID)
  private_spreadsheet_url  TEXT

sheets_target_countries
  id              INTEGER PK
  target_id       INTEGER FK → sheets_targets

sheets_click_tracking
  id              INTEGER PK
  country         TEXT
  tab_name        TEXT
  spreadsheet_id  TEXT
  job_id          INTEGER FK → jobs (nullable, for job-level clicks)
  click_type      TEXT ("tab_navigation" | "job_posting")
  user_identifier TEXT (session/IP hash)
  user_agent      TEXT
  clicked_at      TEXT (ISO datetime)
  referrer        TEXT
  week_id         TEXT

INDEXES: on market_id, canonical_hash, posted_date, status, job_id, click_type,

MIGRATIONS: Applied idempotently at startup via run_migrations().
  001_init.sql                — base schema

9. AI / LLM INTEGRATION
================================================================================

Two AI use cases, both using the Groq API (or xAI Grok as alternative).

--- A. Staging Classification (src/ai/grok_staging.py) ---

When: Called before Google Sheets upload to classify jobs as IT or Non-IT.

How:
  - Reads pending rows from sheets_staging.
  - Batches into chunks of 25 rows.
  - Each row includes: staging_id, title, company, location, country,
    remote_type, normalized_title, current assigned_tab.
  - Sends prompt to Groq asking for IT / Non-IT / Software Engineering /
    Data Science / etc. classification.
  - Multi-key pool: rotates through GROQ_API_KEYS, tracks per-key cooldown.
  - On success: updates assigned_tab in sheets_staging.
  - On failure / unclear: local fallback using IT_KEYWORDS set (regex match
    on title + normalized_title).

IT_KEYWORDS: engineer, developer, software, data, machine learning, ml, ai,
             devops, cloud, analyst, architect, backend, frontend, full stack,
             security, sre, python, java, sql.

Model: llama-3.3-70b-versatile (via api.groq.com/openai/v1)
Rate limiting: per-key cooldown tracking, max(12, len(key_pool)*8) total attempts.

--- B. Quality Enhancement (web_viewer.py — _ai_enhance_quality_rows) ---

When: Triggered via POST /api/jobs/quality/analyze from the admin UI.

How:
  - Reads low-quality job rows (weak country, bad title, etc.) from jobs table.
  - Batches into chunks of 8 rows.
  - Each chunk is sent to Groq to improve: title, normalized_title, country,
    location, remote_type.
  - Multi-worker parallel processing: up to min(len(api_keys), 6) threads.
  - Retry logic (deferred_retries_per_chunk=3, max_retries=4):
    * On 429: parses Retry-After header, sleeps, requeues chunk.
    * On other error: requeues with exponential backoff.
    * Works in both single-worker (serial) and multi-worker (parallel) modes.
  - Suggested fixes stored in memory; user reviews and applies via
    POST /api/jobs/quality/apply.

Model: same as above (GROQ_API_KEY pool).
Max description chars per job: 1400 (truncated to fit token limits).

10. GOOGLE SHEETS INTEGRATION
================================================================================

Enabled by SHEETS_ENABLED=true in .env.
Auth: Google Service Account JSON key file (path = GOOGLE_SA_JSON_PATH).
Scope: https://www.googleapis.com/auth/spreadsheets (read/write).

--- Country Spreadsheets ---

Each active "target" in sheets_targets has one Google Spreadsheet.
Tabs within each spreadsheet are created dynamically per job type (normalized_title).
Each tab has these columns:
  link_id, title, company, location, country, remote_type, posted_date, url

The "url" column contains a tracking URL pointing to the Apps Script web app,
NOT the original job URL. Format:
  {TRACKER_DEPLOYMENT_BASE_URL}?token={TRACKER_TOKEN}&id={staging_id}&
    country={country}&tab={tab}&sheet={sheet_id}&url={encoded_original_url}

--- Tracker Spreadsheet (TRACKER_SPREADSHEET_ID) ---

Central spreadsheet with these tabs:

  Directory   — All uploaded jobs across all countries. Columns:
                doc_key, country, tab_name, link_id, title, company, location,
                apply_url, tracking_url, clicks.
                "clicks" is updated by Google Apps Script on each job click.

  Docs        — Spreadsheet registry. Columns:
                doc_key, spreadsheet_id, country_name.
                Synced automatically on every export run.
                Source of truth for the Apps Script (to look up spreadsheets).

  Clicks      — Tab navigation click log (from /tracker/click endpoint).

  CountryTotals — Aggregated click counts per country.

--- Tracker sync flow ---

Export run (src/reports/tracker_directory_export.py export_directory()):
  1. Reads sheets_staging WHERE status IN ('staged', 'uploaded').
  2. JOINs to jobs for title/company/location.
  3. Calls load_country_doc_configs(conn) — dynamic DB-driven doc_key lookup.
     Falls back to COUNTRY_DOC_KEYS dict, then _slug_doc_key(country).
  4. Writes/clears Docs tab first (all active targets).
  5. Writes/clears Directory tab with all staged/uploaded jobs.

--- Dynamic country/spreadsheet mapping ---

Managed via sheets_targets + sheets_target_countries tables.
  - Admin UI at /admin/sheets_targets allows adding new targets.
  - Each target can have multiple countries mapped to it.
  - doc_key (e.g. "ca", "uk", "germany") is stored per country in DB.
  - Exporter and uploader both use load_country_doc_configs() for consistent
    doc_key resolution — no hardcoded country lists in the code.

--- Click Tracking (Apps Script side) ---

The Apps Script web app (deployed at TRACKER_DEPLOYMENT_BASE_URL):
  - Receives GET requests with token, id, country, tab, sheet, url params.
  - Validates the token matches TRACKER_TOKEN.
  - Increments the "clicks" column in Tracker→Directory for the matching row.
  - Redirects the user to the original job URL.

The Flask app also has a /tracker/redirect route that handles the same
click redirect server-side (records click in sheets_click_tracking).


================================================================================
11. CLICK TRACKING SYSTEM
================================================================================

Two click types are tracked:

1. tab_navigation  — User clicks a country link in the Tracker's Overview tab.
   Producer: Apps Script web app → Tracker Director tab "clicks" column.
   Also recorded in sheets_click_tracking (click_type='tab_navigation').

2. job_posting     — User clicks the "Apply" tracking URL for a specific job.
   Producer: /tracker/redirect in Flask OR Apps Script intercept.
   Recorded in sheets_click_tracking with job_id populated.
   The Tracker→Directory "clicks" column is incremented by Apps Script.

The sheets_click_tracking table records:
  country, tab_name, spreadsheet_id, job_id, click_type,
  user_identifier (session/IP hash), user_agent, clicked_at, referrer, week_id.

Click data is NOT currently synced back from Google Sheets to the local DB
(the "clicks" counter in Tracker→Directory is the live source of truth in Sheets).


================================================================================
12. ANALYTICS ENGINE
================================================================================

All analytics modules live in src/analytics/ and read from the SQLite DB.

weekly_metrics.py
  - Determines ISO week boundaries (Monday start, ISO 8601).
  - Queries skills+jobs for current week and N weeks prior.
  - Computes frequency (count of jobs mentioning each skill).
  - growth_percentage = (current - prior) / prior × 100.
  - absolute_delta = current - prior.
  - mover_score = absolute_delta × log1p(frequency) — penalizes low-base spikes.
  - Calls emerging_detector.assign_flags() to mark each skill.
  - Writes WeeklyMetric rows to weekly_metrics table.

emerging_detector.py
  - emerging_flag: growth ≥ GROWTH_THRESHOLD (default 20%) AND freq ≥ MIN_FREQ.
  - declining_flag: growth ≤ DECLINING_THRESHOLD (default -15%).

co_occurrence.py
  - Computes which skill pairs appear together in the same job description.
  - Used by /api/skills/<name>/co-occurring endpoint.

temporal_trends.py
  - Multi-week rolling trends per skill.

title_analytics.py
  - Job title distribution, normalized title frequencies.

category_analytics.py
  - Skill category breakdown (languages, frameworks, cloud, etc.).

coverage_metrics.py
  - Overall stats: total jobs, remote %, source breakdown, geo distribution.

Analytics are computed once per pipeline run and cached in weekly_metrics table.
The dashboard and API endpoints read pre-computed metrics from the DB directly.

  Daily:  python -m src.orchestrator --mode ingest-only  (07:00)

17. HOSTING CONSIDERATIONS
================================================================================

CURRENT STATE: Runs entirely on a local Windows machine.

KEY CONSTRAINTS FOR HOSTING:

1. DATABASE
   The database is a single SQLite file (data/jobs.sqlite). SQLite works well
   for single-process read/write but is NOT suitable for multi-process or
   high-concurrency web hosting (e.g., multiple Gunicorn workers writing
   simultaneously will cause locking issues).

   HOSTING OPTIONS:
   a) Single-process deployment (1 worker): SQLite works fine.
      Good for: VPS, Railway, Render (single instance), fly.io.
   b) Multi-process: Migrate to PostgreSQL or MySQL.
      The schema uses standard SQL; migration is feasible but requires
      adapting db.py (replace sqlite3 with psycopg2/SQLAlchemy).

2. FILE SYSTEM PERSISTENCE
   The project writes to:
     data/jobs.sqlite          — database
     outputs/{market}/{week}/  — reports, CSVs, PNGs
     logs/{week}/              — log files
   Cloud platforms with ephemeral file systems (Heroku, Render free tier)
   will lose these on restart. Use a persistent volume or external storage.

3. ENVIRONMENT SECRETS
   The .env file contains API keys and the Google Service Account JSON path.
   On a hosted platform, set these as environment variables in the host's
   dashboard (never commit .env or the service account JSON to git).

4. GOOGLE SERVICE ACCOUNT JSON
   The file config/job-market-intelligence-489015-57c9087db0cf.json is a
   private credential file. For hosting:
   - Store its contents as an environment variable (base64-encoded) or
     mount it as a secret file via the hosting platform's secrets management.
   - Update GOOGLE_SA_JSON_PATH to point to the mounted path.

5. FLASK FOR PRODUCTION
   The current setup uses Flask's built-in development server (debug=False
   but still single-threaded). For production hosting:
   - Use Gunicorn: gunicorn -w 1 -b 0.0.0.0:5000 web_viewer:app
     (1 worker = SQLite safe)
   - Or uWSGI with 1 process.
   - Add a reverse proxy (nginx / Caddy) in front for TLS, caching, etc.

6. PIPELINE SCHEDULING
   Currently uses Windows Task Scheduler. On a Linux server/cloud:
   - Use cron for periodic pipeline runs.
   - Or a process manager like supervisor to keep web_viewer.py alive.
   - The schedule library is also built in (src/run_manager.py) for
     in-process scheduling, but a separate cron job is cleaner.

7. RECOMMENDED MINIMAL HOSTING SETUP (LOW COST)
   Platform: fly.io, Railway, or a $5/month VPS (DigitalOcean, Hetzner).
   Setup:
     a) Deploy Flask app as a single-process Gunicorn container (1 worker).
     b) Mount a 1–5 GB persistent volume at /data/ for jobs.sqlite.
     c) Set all .env values as platform environment variables.
     d) Mount service account JSON as a secret file + set GOOGLE_SA_JSON_PATH.
     e) Add a cron job (fly cron / Railway cron / system cron) to run:
          python -m src.orchestrator --mode weekly
        once per week (Monday morning).
     f) The web dashboard becomes publicly accessible at your deployed URL.
        Set WEB_VIEWER_URL to that URL.
        Update TRACKER_DEPLOYMENT_BASE_URL to reflect the new host so
        job click tracking URLs remain valid.

8. WHAT DOES NOT NEED HOSTING
   - Google Sheets (external — already hosted by Google).
   - Google Apps Script click tracker (external — already deployed).
   - API data sources (all external).
   Only the Flask web app + SQLite DB + scheduled pipeline need a server.

9. DATA SIZE ESTIMATE
   Each job row ~3–5 KB (with description text).
   At 500 jobs/day ingested: ~1.5 MB/day raw.
   With deduplication (~40–60% reduction): ~0.6–0.9 MB/day net growth.
   After 1 year: ~200–300 MB database.
   Skills table: ~10–20x job count rows (small).
   Outputs directory: small (text files + PNGs, <10 MB/week).
